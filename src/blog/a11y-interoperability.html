Accessibility

How do people with disabilities use the internet? Most blind people do not use a mouse. Instead, they use keyboard input to operate a screen reader. People who are deafblind might use a screen reader with a refreshable Braille display. Some people use neither a mouse nor a keyboard and instead use their voice to issue commands to speech recognition software.

These alternative ways of navigating the web rely on properly marked up HTML code.

For example, marking up all headings using heading elements (h1-h6) will make them available when a screen reader user navigates via the heading outline.

When a user focuses an element, it will announce the type of element, and also any relevant state properties, such as being disabled. Some of these state properties are defined by ARIA attributes, which can be added to and removed from elements in order to indicate state changes and relationships to other elements.

Interoperability

We should also consider the ability for external sites and services to gather information from the page for indexing and sharing.

Audio-enabled digital assistants, for instance, often parse web pages' speakable summaries, document outlines and structured data to find information. The relevant data it finds is packaged into an experience that is native to that service's voice UI.

Browsers may have plugins or provide features, such as reader view, that parse web pages, so it is best to provide as much relevant metadata as possible and always adhere to the HTML specification.

Engineering Strategies

The rules of accessibility and interoperability can be complex and cumbersome. Some kind of automation needs to be in place in order to keep application code simple and bug-free.

Creating a library of reusable web components is the best way to build an accessible, interoperable web site.

There also needs to be a mechanism for creating and tracking IDs to be used by the ARIA attributes aria-labelledby and aria-describedby.

Glossary of Terms for Accessible and Interoperable HTML

Traditional End User Technologies

Vocabulary Terms for

  • Graphical User Interface

    The current state of the document tree is rendered into a visual representation and refreshed 60 times a second. People who aren't using assistive technologies read and navigate the web using a graphical user interface.

  • Fine Pointer Input

    The page can be scrolled and controls can be activated with a fine (precise) pointing device, typically a mouse controlled with the dominant hand and fingers.

  • Coarse Pointer Input

    The touch of a finger is treated as a coarse (imprecise) pointing device. The page can be scrolled by swiping and controls can be activated by tapping.

  • Keyboard Input

    Web pages are natively navigable with keyboard input. The page can be scrolled with the space bar and controls can be tabbed to and then activated with the enter key.

  • Keyboard Focus

    At all times, one element on a web page will be marked as being in focus. Focus acts like a virtual cursor in a text document.

  • Assistive Technology (AT)

    An assistive technology is any device or feature that is designed to facilitate interaction between a computer and a user with a disability. Screen readers, refreshable Braille displays, and speech recognition software are three of the most common types of AT.

  • Screen Reader

    A screen reader is software that uses a synthesized voice to announce the labels and properties of elements on a web page. Screen readers interact with the accessibility tree, a textual representation of the user interface.

  • Full Access Keyboard Navigation

    Screen readers typically enable full access keyboard navigation, where both interactive and non-interactive items are navigable with the arrow keys. Screen readers also provide several keyboard shortcuts for navigating the text-based UI, such as a mode for navigating only headings or only links.

  • Refreshable Braille Display

    A refreshable Braille display is a device that is used in conjunction with a screen reader to read a web page's text-based accessibility tree through touch instead of through speech synthesis.

  • Speech Recognition Software

    Speech recognition software accepts voice commands to do things like scroll the page, click links and buttons, and enter text.

  • Annotated Graphical User Interface

    Voice Recognition Software may allow selection of items using their accessible label. Additionally, unlabelled or mislabeled items can be selected using an annotated visual overlay with numbers or letters for easy identifcation.

  • Reader View

    Reader View is a built-in or third-party browser feature that examines a web page's structure to find its main content and renders it in a custom layout.

  • Browser Plugins

    Third-party browser plugins have full access to web page markup, including metadata. For maximum interoperability, developers should maintain spec-compliant markup and provide useful structured data.

Recent and Emerging End User Technologies

Vocabulary Terms for

  • Digital Assistant

    Digital assistants parse voice queries and respond in a synthesized voice. These services often query the web pages in order to find answers.

  • Voice User Interface

    A voice user interface, also known as a conversational UI, operates as a two-way dialogue between a human and a service using natural language.

  • Native Search Results

    Search engines and digital assistants may copy a relevant bit from a web page and embed it as a native search result, for example as an answer or a widget, without requiring the user to visit the web page directly.

  • Speakable Web Pages

    When a digital assistant determines that a web page should be read aloud, it will look for metadata which labels certain elements as speakable areas.

Developer Technologies

Vocabulary Terms for

  • Document Tree

    The document tree, or Document Object Model (DOM), is a representation of the nested markup of a web page in its current state.

  • ARIA

    Accessible Rich Internet Applications (ARIA) attributes can be added to elements to indicate state changes, relationships, and roles (types) to assistive technologies.

  • Accessibility Tree

    The accessibility tree is a stripped-down representation of the DOM that is derived from elements' roles and ARIA attributes. Screen readers interact with this tree (the AOM) instead of the DOM.

  • Structured Data

    Structured data is a representation of the DOM that is derived from microdata associated with certain elements, or from JSON-LD. When search engines and digital assistants find relevant bits of structured data on a web page, they may copy it and present it to the user in a native experience.

  • Heading Outline

    A heading outline is a representation of the DOM that is derived from heading levels. It is a navigation tool for screen reader users, who use it to skim through long pages quickly.

  • Document Outline

    The document outline is a representation of the DOM that is derived from sectioning content. Vendors may detect sectioning elements to derive the structure of a document (for reader view, for example). However, it is not exposed to the accessibility tree, so it cannot be used as a navigation tool for screen reader users.

  • HTML Spec Compliance

    Compared to invalid markup, spec-compliant markup is more likely to work consistently across browsers and assistive technologies. According to the HTML specification, elements are only valid inside certain other elements. A <fieldset> isn't valid inside a <button>, for example. Elements also have restrictions on which ARIA roles can be applied to them.

  • Reader View Heuristics

    When vendors attempt to identify the main content of a web page, they use a mix of the document outline, structured data, and other factors based on common HTML usage.

    Without any paragraph tags, for example, Reader View may reject a web page. When the content of only one section is marked up with paragraph tags, Reader View may select only that section to display.
